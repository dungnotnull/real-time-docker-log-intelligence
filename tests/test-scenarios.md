# Test Scenarios — Dozzle-Enhanced AI Log Analysis Agent

## Scenario 1: OOM Kill Anomaly Detection
**User trigger:** Container `api-server` is killed by Linux OOM killer. Kernel writes "Killed process 1234 (node)" to Docker logs.

**Expected behavior:**
1. DozzleClient receives the OOM log line from Docker socket
2. BGE-large embeds the line; Isolation Forest scores it below -0.3 (high anomaly)
3. Anomaly event stored in `anomaly_events` table with `anomaly_score < -0.3`
4. ErrorExplainer batch triggered after 60-second window flush
5. Claude API generates explanation: `error_type=OutOfMemoryKill, severity=CRITICAL, fix_steps=[increase memory limit, profile for leaks]`
6. If webhook configured: POST to Slack with severity=CRITICAL

**Assertions:**
- `anomaly_score < -0.3` for the OOM log line
- `explanation.error_type == "OutOfMemoryKill"` 
- `explanation.severity == "CRITICAL"`
- Explanation stored in `anomaly_events.explanation_json`
- Webhook called (if WEBHOOK_URL set)

**Test file:** `test_agent.py::TestAnomalyDetector::test_oom_detection`

## Scenario 2: Natural Language Log Query
**User trigger:** DevOps engineer types: "show me postgres connection errors from the last 30 minutes"

**Expected behavior:**
1. NLLogQuery receives query string
2. MiniLM retrieves closest template (connection errors)
3. LLM (Claude) parses: `{container_name: "postgres", time_range_minutes: 30, log_level: "ERROR", keywords: ["connection"]}`
4. SQLite filter executed: `WHERE container LIKE '%postgres%' AND ingested_at >= (now-30min) AND log_level='ERROR' AND message LIKE '%connection%'`
5. Returns paginated results (up to 100 lines) sorted by timestamp desc
6. Result cached in `nl_query_cache` (5-min TTL)

**Assertions:**
- `parsed_filter["container_name"]` contains "postgres"
- `parsed_filter["time_range_minutes"] == 30`
- `parsed_filter["log_level"] == "ERROR"`
- Results returned in < 3 seconds
- Second identical query returns from cache (no LLM call)

**Test file:** `test_agent.py::TestNLLogQuery::test_nl_query_parse`

## Scenario 3: Novel Log Cluster Detection
**User trigger:** New microservice deploys with a unique error format never seen before: "CIRCUIT_BREAKER_OPEN: downstream-payments tripped after 5 failures"

**Expected behavior:**
1. BGE-large embeds the log line
2. LogClusterer compares to all existing cluster centroids via FAISS
3. Max cosine similarity < 0.75 → creates new cluster (cluster_id=N, is_novel=True)
4. Novel cluster queued for LLM label generation
5. `is_novel_cluster=True` flag returned to orchestrator
6. Anomaly event recorded with `is_novel_cluster=True`

**Assertions:**
- New cluster created with `is_novel_cluster=True`
- Cluster size = 1 on first occurrence
- LLM label generated: 3-5 words describing the circuit breaker pattern
- Second occurrence of same message assigned to same cluster (similarity > 0.75)

**Test file:** `test_agent.py::TestLogClusterer::test_novel_cluster_creation`

## Scenario 4: High-Volume Log Throughput
**User trigger:** Stress test — 10,000 log lines/second generated by log-generator container.

**Expected behavior:**
1. DozzleClient reads lines at 10,000/sec rate
2. Batches of 50 lines processed every 100ms
3. BGE-large embedding queue does not fall behind (processes in < 50ms per batch)
4. Isolation Forest scoring completes in < 5ms per batch
5. SQLite writes complete without blocking ingest queue
6. No `asyncio.QueueFull` exceptions in 60-second sustained run

**Assertions:**
- `stats["lines_processed"]` increments at ≥ 10,000/sec
- Memory stays < 500MB additional during sustained load
- Zero `QueueFull` exceptions in log
- P95 processing latency < 200ms per batch

**Test file:** `test_agent.py::TestOrchestrator::test_high_volume_throughput`

## Scenario 5: LLM Provider Fallback
**User trigger:** ANTHROPIC_API_KEY is valid but Claude API returns 429 (rate limit) three times in a row.

**Expected behavior:**
1. LLMClient calls Claude, receives 429
2. Exponential backoff: wait 1s → 2s → 4s
3. After 3 failed Claude attempts, falls back to OpenAI gpt-4o
4. OpenAI call succeeds; explanation returned
5. LLM cost logged with `provider=openai`
6. No user-visible error

**Assertions:**
- Total retries = 3 before fallback
- Final provider used = "openai"
- `llm_cost_log` entry has `provider="openai"`
- `explain()` returns valid explanation dict

**Test file:** `test_agent.py::TestLLMClient::test_provider_fallback`

## Scenario 6: LSTM Sequence Anomaly (Gradual Degradation)
**User trigger:** Redis container slowly degrades over 5 minutes: latency warnings increase in frequency, then ERROR level messages appear. Pattern: INFO → INFO → INFO → WARN → WARN → ERROR → ERROR → CRITICAL (repeated across 20-line windows).

**Expected behavior:**
1. LSTM Autoencoder processes 20-line sliding window
2. First 50 windows (normal): reconstruction error stays < threshold
3. Window 51 onwards (degradation): reconstruction error exceeds mean + 2σ
4. `is_seq_anomaly=True` returned for the window containing the degradation onset
5. ErrorExplainer triggered; Claude explains: "Gradual service degradation pattern detected, likely resource exhaustion"

**Assertions:**
- No false positives during first 50 normal windows
- `is_seq_anomaly=True` triggers within first 3 degradation windows
- LSTM threshold dynamically adapts (mean + 2σ of last 1000 windows)
- Explanation includes "gradual" or "increasing" pattern description

**Test file:** `test_agent.py::TestAnomalyDetector::test_lstm_sequence_anomaly`

## Scenario 7: Full E2E Incident Response
**User trigger:** Production incident — 3 containers fail in 2 minutes: `postgres` (connection refused), `api-server` (database errors cascade), `nginx` (502 Bad Gateway).

**Expected behavior:**
1. All 3 container log streams simultaneously show anomalies
2. Anomaly detector flags 15+ anomalous lines across 3 containers
3. Clusterer groups: `DB_CONNECTION_FAILED` cluster (postgres + api-server), `GATEWAY_ERROR` cluster (nginx)
4. ErrorExplainer batches: 10 lines from api-server, 5 from nginx, 5 from postgres
5. Claude generates 3 separate explanations + 1 cross-service incident summary
6. CLI command `explain --container api-server` returns root cause: "postgres connection refused (upstream failure)"
7. Webhook sends CRITICAL alert within 90 seconds of incident start

**Assertions:**
- All 3 containers' anomalies detected within 30 seconds
- Cross-container cluster correlation identifies shared root cause
- Incident summary references all 3 affected services
- CLI explain returns description linking api-server errors to postgres
- Webhook alert fired with `severity=CRITICAL`
- Total pipeline latency (first anomaly → explanation) < 90 seconds

**Test file:** `test_agent.py::TestIntegration::test_cascade_incident_response`
